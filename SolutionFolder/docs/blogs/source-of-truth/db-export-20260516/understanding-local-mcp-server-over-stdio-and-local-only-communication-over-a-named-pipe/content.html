When integrating systems inside Visual Studio, two communication patterns often appear early in the design:

  1. A local MCP server running over stdio
  2. Local-only communication over a named pipe

At first glance, these can seem interchangeable. In practice, they solve different problems and are often used together in a layered architecture.

Local MCP Server Over stdio

A process running “over stdio” communicates through its standard input and output streams. In .NET, this is typically done by launching a child process and redirecting its input and output streams.

Microsoft documents this capability through System.Diagnostics.Process, where standard input and output can be redirected and used programmatically: Process.StandardInput documentation .

This allows a host process to send requests and receive responses without opening network ports or configuring external communication channels.

Conceptual model

[ Host / Bridge Process ]  <-- stdin/stdout -->  [ MCP Server Process ]

In this model:

  • The host launches the MCP server as a child process.
  • Requests are written to the server’s standard input.
  • Responses are read from standard output.

This approach is simple and predictable. It works well when one process owns the lifecycle of another and needs a direct communication channel.

Example

var startInfo = new ProcessStartInfo
{
    FileName = "VsMcpBridge.McpServer.exe",
    RedirectStandardInput = true,
    RedirectStandardOutput = true,
    UseShellExecute = false
};

using var process = Process.Start(startInfo);
using var writer = process.StandardInput;
using var reader = process.StandardOutput;

await writer.WriteLineAsync("{\"method\":\"ping\"}");
var response = await reader.ReadLineAsync();

On the server side, communication is equally direct:

while (true)
{
    var line = Console.ReadLine();
    if (line == null)
    {
        break;
    }

    Console.WriteLine("{\"result\":\"pong\"}");
}

The key idea is that stdio is tightly coupled to a parent/child process relationship.

Local-Only Communication Over a Named Pipe

A named pipe is an operating system–managed communication channel designed for interprocess communication. In .NET, this is implemented using NamedPipeServerStream and NamedPipeClientStream.

Microsoft describes named pipes as a mechanism for communication between processes, where a server can support one or more clients: NamedPipeServerStream documentation .

Unlike stdio, named pipes are not tied to a parent/child relationship. Any authorized process on the same machine can connect to the pipe.

Conceptual model

[ Visual Studio Extension ]  <-- named pipe -->  [ Local Bridge / Service ]

In this model:

  • The server creates a named pipe.
  • Clients connect to that pipe by name.
  • Both sides exchange messages over a bidirectional channel.

Server example

using var server = new NamedPipeServerStream("VsMcpBridge");

server.WaitForConnection();

using var reader = new StreamReader(server);
using var writer = new StreamWriter(server) { AutoFlush = true };

var request = reader.ReadLine();
writer.WriteLine("ack");

Client example

using var client = new NamedPipeClientStream(".", "VsMcpBridge");

client.Connect();

using var reader = new StreamReader(client);
using var writer = new StreamWriter(client) { AutoFlush = true };

writer.WriteLine("hello");
var response = reader.ReadLine();

By default, the pipe direction supports bidirectional communication, making it well-suited for request/response interactions.

Key Differences

  • stdio: Direct communication between a parent process and the child process it launches.
  • named pipe: A reusable, OS-managed channel for communication between local processes.

Each has a clear role depending on how processes relate to each other.

Combining Both Patterns

These two approaches are often used together in a layered design:

[ Visual Studio Extension ]
          |
     named pipe
          |
   [ Local Bridge ]
          |
        stdio
          |
   [ MCP Server Process ]

In this structure:

  • The extension communicates locally with a bridge using a named pipe.
  • The bridge manages the MCP server process using stdio.

This separation keeps responsibilities clear:

  • The extension focuses on integration with Visual Studio.
  • The bridge handles process management and protocol translation.
  • The MCP server focuses on domain-specific behavior.

Takeaway

The distinction can be summarized simply:

  • stdio is ideal for controlling and communicating with a launched child process.
  • named pipes are ideal for structured communication between local processes.

Understanding how and when to use each makes system design more deliberate and significantly easier to reason about.

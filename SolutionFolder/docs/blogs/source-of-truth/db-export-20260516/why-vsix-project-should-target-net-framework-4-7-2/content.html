When working with Visual Studio extensions, one detail that is easy to miss is that the extension project must align with the runtime used by the Visual Studio host. For Visual Studio 2022, that means targeting .NET Framework 4.7.2.

Microsoft’s guidance for in-process extensions makes this requirement explicit: In-proc extensions must target the .NET version used by the Visual Studio version they run in .

Why this matters

A VSIX extension does not run as an independent application. It is loaded directly into the Visual Studio process. Because of this, it must match the runtime expectations of the Visual Studio shell and the VSSDK.

For Visual Studio 2022, that runtime is .NET Framework 4.7.2. Attempting to treat the extension project like a modern standalone application (for example, targeting .NET 6 or later) will lead to incompatibilities in loading, packaging, or execution.

Clarifying the architecture

Understanding this constraint helps separate responsibilities clearly:

  • VSIX extension project — must follow Visual Studio’s in-process runtime model.
  • External components — can use modern .NET runtimes if they run out-of-process.

This separation allows the extension to remain stable and compatible, while still enabling newer technologies where they are appropriate.

Practical implications

In practice, this leads to a few important guidelines:

  • Target .NET Framework 4.7.2 for VSIX projects targeting Visual Studio 2022.
  • Do not attempt to force SDK-style or modern runtime assumptions into the extension host.
  • Keep newer runtime usage isolated to companion processes or services.

Lifecycle considerations

This runtime alignment also reflects a broader principle: Visual Studio owns the lifecycle of extension components.

For example, tool windows are not constructed through application-controlled dependency injection. Instead, they are created by the shell, and initialization should occur at the appropriate lifecycle point: ToolWindowPane.OnToolWindowCreated() .

Recognizing this pattern helps avoid common pitfalls, such as attempting constructor injection where the shell controls instantiation.

Takeaway

For Visual Studio 2022 extensions, targeting .NET Framework 4.7.2 is not a legacy constraint—it is a requirement of the hosting environment.

Once that constraint is understood, the rest of the extension architecture becomes clearer, more predictable, and easier to reason about.

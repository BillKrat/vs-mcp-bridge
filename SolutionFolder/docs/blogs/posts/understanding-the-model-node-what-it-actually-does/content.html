Understanding the Model Node

What It Actually Does (and Doesn't Do)

Here is a natural, reasonable mental model of "chat with your document": you upload a file, the model reads it, then you ask questions and the model answers from what it read.

Almost none of that is what actually happens. Walking through it concretely is the fastest way to see how much less the Model itself is doing than it feels like from the outside — and why that gap is exactly why systems like this separate Model from Retrieval, Memory, and Agent instead of treating "the AI" as one thing.

What Happens When You Upload a Document

Nothing involving the Model. Specifically, in order:

  1. If the file is a binary (PDF, .docx), its text gets extracted first — a parsing step, not a Model step.
  2. The extracted text gets chunked — split into paragraphs or fixed-size pieces.
  3. Each chunk gets converted into a vector (an embedding). This step does involve a trained neural network doing inference — but typically a separate, smaller, specialized embedding model, not the same large generative model you chat with.
  4. Those chunk-vectors go into a vector store. The original chunk text is also kept somewhere retrievable.

At the end of that sequence, the Model has done nothing. It hasn't seen the document. It doesn't know it exists.

What Happens When You Ask a Question

  1. Your query gets embedded into a vector too (the same embedding-model step as above).
  2. That query vector gets compared against the stored chunk-vectors — a similarity search. This is math run against a precomputed index. No large model is involved in this step at all.
  3. The top-matching chunks come back out as text — the vectors were only ever the search mechanism.
  4. Your query, the retrieved chunks, and any system instructions get assembled into one prompt.
  5. Only now does the Model do anything: generate a response from whatever text just landed in its context window.

This is the standard retrieval-augmented generation pattern (Lewis et al., 2020). It's also exactly the pattern challenged by recent Direct Corpus Interaction (DCI) research, which argues that pre-chunking a corpus into a fixed top-k candidate set can bottleneck a capable agent that could instead search and read the corpus directly. Different proposed architecture, same underlying point: retrieval and generation are separable concerns, not one blended "the AI did it."

Two Steps That Look Alike But Aren't

The similarity search (step 2 above) and the Model's generation step feel similar from outside — both take an input and produce a relevant-looking output. Mechanically they're different categories of work:

Similarity Search Model Generation
What it operates on A precomputed, stored vector index Whatever text is currently in the context window
What it does Finds the closest existing match by distance Synthesizes new text, token by token
Is it "reasoning"? No — fixed math, no learned judgment in the moment Yes — runs the input through trained weights

The Misconception Worth Naming Directly

It's natural to assume the Model, when generating, is somehow consulting what it was trained on — "looking things up" in its training data the way the similarity search looks things up in a vector index.

It isn't, and it can't. The Model has no access to its training data at inference time, at all. Training data did its job once, during training: it shaped a large set of numbers (weights). Once training is done, those weights are what's left — a fixed function. When you talk to the Model, it isn't searching anything; it's running your current context through that already-shaped function. Nothing about the training data itself is present or reachable at that moment. This is also the precise, mechanical reason systems ground models in retrieval at all: since the Model can never look anything up on its own, anything it needs to know that isn't baked into its weights has to be handed to it directly, inside the context window, every single time.

So Who Assembles the Context Window?

Not the Model — it only ever receives an already-built prompt. And, somewhat counter-intuitively, not memory/storage either. A memory component is a container: it holds conversation history and long-term state, and answers with what it has when asked. It doesn't decide what should go into this particular prompt, how much of it, or in what order — that's a decision, not stored knowledge, and deciding is a different kind of responsibility than holding. That decision belongs to whatever is directing the overall task — the orchestration/agent layer: pull from memory, pull from retrieval results, add instructions, combine them into one prompt sized to fit.

Which leads to a genuinely useful piece of intuition: a smaller context window doesn't mean information gets "forgotten" so much as it means there's less room, so material competes for the same limited space and older or less-relevant pieces get crowded out by newer ones. Nothing is erased; it's just no longer included in what got assembled this time.

Takeaway

The Model's real job is narrow: take whatever text it's handed, generate a response. It doesn't touch files, doesn't search anything, doesn't remember anything between calls, and doesn't consult its own training data while running. Everything that feels like "the AI understanding my document" is actually a chain of separate, mundane steps — extraction, chunking, embedding, storage, search, assembly — each owned by a different part of the system, handing the Model exactly and only what it needs for this one call.

Seeing that chain clearly is what makes the rest of the system's architecture make sense: retrieval, memory, and orchestration aren't extra complexity bolted onto "the real AI." They're the parts doing most of the actual work, around a Model that does one thing well and nothing else.

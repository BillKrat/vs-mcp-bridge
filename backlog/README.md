# Backlog

## Purpose

The backlog is the intake area for new ideas, research items, future improvements, architecture possibilities, and shiny objects.

Rule:

> Capture immediately, prioritize later. Creating a backlog item is not authorization to start work.

## Operating Rule

Active execution must remain focused on the current prioritized beta-release path.

New ideas must be captured here unless they are required to complete the currently active slice. A captured idea may be reviewed, refined, grouped, split, closed, or prioritized later.

Adding an item to this backlog does not permit implementation. It does not authorize runtime code, deployment, route changes, GitHub project changes, automation, scripts, or tool work.

## Who This Applies To

This backlog rule applies equally to:

- human contributors
- ChatGPT
- Codex
- Copilot
- Cursor
- future agents
- future tools

No contributor or tool should redirect active work away from the current prioritized beta-release path without explicit prioritization.

## How To Add An Item

1. Copy `backlog/template.md`.
2. Create a short, descriptive backlog item file in this folder.
3. Fill in enough context to preserve the idea.
4. Return to the active slice.

Keep backlog items practical and bounded. The goal is to preserve useful ideas without letting them interrupt active execution.

## Review And Prioritization

Backlog items may be reviewed later when:

- the active beta-release slice is complete
- repeated friction proves the idea matters
- a human explicitly asks to prioritize backlog work
- the item becomes a blocker for the active path

Until then, backlog items remain captured, not active.
